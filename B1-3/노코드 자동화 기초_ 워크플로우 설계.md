# 마케팅 자동화 파이프라인 구축 결과보고서

## 1. [프로젝트 1] 조건 분기 자동화 (Zapier vs Make)

### ① 자동화 반복 업무 정의

* **업무 정의:** 구글 스프레드시트에 테스트 대상자의 성적 데이터가 입력되면, 점수 기준에 따라 합격자에게는 안내 이메일을 자동 발송하고, 탈락자는 시트에 자동으로 기록을 업데이트하는 조건 분기 시스템 구축

### ② 도구 선정 및 비교

* 동일한 워크플로우를 Zapier와 Make 두 가지 플랫폼에서 각각 구현하여 인터페이스의 직관성, 라우팅 조건 설정의 편의성, 실행 효율성을 비교·검증

| 비교 항목 | Zapier | Make |
| ----- | ----- | ----- |
| **분기 처리 방식** | Paths 기능을 활용한 하향식 트리 구조 | Router 노드를 중심의 방사형 허브 구조 |
| **시각적 직관성** | 단계별 스크롤 방식 (전체 흐름을 한눈에 파악하기 다소 복잡) | 2D 캔버스 기반 (데이터의 흐름과 통과 여부를 직관적으로 확인 가능) |
| **조건 설정 편의성** | 상위 노드의 필드 매핑이 정형화되어 있어 안정적 | 노드 간 변수 드래그 앤 드롭 및 실시간 필터 갱신 기능 제공 |
| **무료 플랜 범위** | 월 100 Tasks | 월 1,000 Operations |
| **실행 로그 확인 방식** | Task History에서 성공/실패 건수와 타임스탬프 위주로 표시 | 시나리오 History에서 각 모듈별 입출력 데이터(Bundle)까지 상세 확인 가능 |
| **연동 서비스 범위** | 대중적 SaaS 커버리지가 넓음 | API/Webhook 커스텀 연동 자유도가 더 높음 |
| **에러 핸들링/재시도** | Auto-replay 기능이 제한적이며 유료 플랜에서 강화됨 | 무료 플랜에서도 Error Handler 라우트를 세밀하게 설계 가능 |

### 도구별 장단점 정리

**Zapier**
* 장점: 인터페이스가 단순해 러닝커브가 낮고, 대중적인 SaaS 앱과의 연동 안정성이 높음
* 단점: 무료 플랜의 Task 한도가 낮고, 복잡한 분기 구조는 스크롤 기반이라 전체 흐름을 한눈에 파악하기 어려움

**Make**
* 장점: 2D 캔버스로 전체 워크플로우를 한눈에 파악할 수 있고, 모듈별 입출력 데이터를 상세히 디버깅할 수 있음
* 단점: 초기 학습 곡선이 있고, Router/Filter 등 개념을 이해하는 데 시간이 더 필요함

### 적합 상황 의견

단순한 선형 워크플로우이고 빠른 셋업과 익숙한 UI가 중요한 경우에는 Zapier가 적합하다. 반면 이번 과제처럼 조건 분기가 여러 개 얽혀 있고 실행 결과를 단계별로 세밀하게 검증해야 하는 경우에는 Make의 캔버스 기반 시각화와 상세 로그가 더 유리하다.

## 2. [프로젝트 2] 자유 주제 자동화 설계 및 구현 (콘텐츠 최적화 파이프라인)

### ① 자동화할 반복 업무 정의

* **업무 정의:** 구글 시트에 마케팅 주제와 희망 채널을 입력하면, AI가 타깃 지향적이고 매력적인 브랜드 스토리텔러로 빙의하여 네이버 블로그 맞춤형 원고 또는 인스타그램 캐러셀 카드뉴스 기획안을 단 한 번에 자동 생성한 후 본인 Gmail로 즉시 전송하는 시스템

### ② 도구 선정 및 선정 이유

* **선정 도구:** **Make (메이크) & OpenAI (ChatGPT) & Gmail**
* **선정 이유:**
  * Make의 라우터 필터 기능을 통해 단 하나의 시트 안에서 블로그와 인스타그램의 타깃 특성에 맞는 개별 프롬프트 엔진을 조건별로 유연하게 제어하기 위함
  * 원고 초안 작성 시간을 줄이기 위해, 가독성과 편집 효율성을 고려한 텍스트 출력을 이메일로 즉시 자동화함 (단, 생성된 초안은 세부 표현이나 톤 조정 등 최소한의 검수·수정을 거친 뒤 실무에 활용하는 것을 전제로 함)

### ③ 워크플로우 설계 문서 (흐름 설명)

1. **Trigger (구글 시트):** 새 행이 추가되면 입력된 주제 키워드와 채널 종류를 실시간 감지함.
2. **Router (조건 필터 분기):**
   * **1st 블로그 포스팅 경로:** 채널 종류가 '블로그'와 정확히 일치할 때 통과
   * **2nd 인스타그램 카드뉴스 경로:** 채널 종류가 '인스타그램'와 정확히 일치할 때 통과
3. **Action (AI 원고 생성 및 발송):**
   * 각 경로에 내장된 맞춤형 프롬프트(되묻기 단계 배제, 모바일 줄바꿈 최적화, 후킹 제목 공식 내장)를 기반으로 OpenAI가 원고를 한 번에 연산한 후, 본인의 Gmail로 자동 전송함.

![프로젝트2 워크플로우 셋업](../images/B1-3/project2_workflow_setup.png)

### ④ 채널별 최종 실행 결과 산출물

#### [블로그 채널 테스트 결과]

* **입력 주제:** 장마철 건강 관리
* **출력 원고 파싱:** 인사말을 배제한 질문형 도입부, 모바일 가독성 가이드라인(20~35자 내외 줄바꿈), 실제 경험 사례 중심의 공감 스토리텔링이 반영된 마크다운 기반 원고가 전달됨. 다만 문장 톤과 세부 표현은 실무 투입 전 검수·수정이 필요함.

![블로그 최종 결과 이메일](../images/B1-3/project2_result_output1.png)

#### [인스타그램 채널 테스트 결과]

* **입력 주제:** 소고기 장조림 
* **출력 원고 파싱:** 100가지 후킹 제목 라이브러리 중 최적 후보군을 선별(`퇴근 후 30분만에 만드는 소고기 장조림 6단계`)하고, 표지(1P)부터 요약(9P), 저장 유도 CTA(10P)까지 이미지 컨셉 가이드를 포함한 캐러셀 구성안 및 피드 본문 캡션이 수신됨. 다만 카피 문구와 이미지 컨셉 가이드는 그대로 발행하기보다 브랜드 톤에 맞춰 다듬는 작업이 필요함.

![인스타그램 최종 결과 이메일](../images/B1-3/project2_result_output2.png)

#### [한계점 및 실무 적용 시 유의사항]

* AI가 생성한 블로그 원고와 인스타그램 캐러셀 구성안은 초안(draft) 수준으로, 톤앤매너·세부 표현·이미지 컨셉을 브랜드 가이드에 맞게 수정하는 최소한의 검수 과정을 거친 뒤 발행하는 것을 전제로 함
* 즉, 이 자동화 파이프라인의 핵심 가치는 "완성 원고 자동 생성"이 아니라 "초안 작성 시간 단축 및 아이디어 초기화(idea kick-start)"에 있음

## 3. 보안 및 과금 리스크 관리 서약

* **보안 준수:** 본 결과보고서에 첨부된 모든 캡션 및 스크린샷 내의 개인 구글 계정 주소, Gmail 수신자 메일, 스프레드시트 고유 키 값 및 API 인증 세부 정보는 마스킹 처리하여 외부 노출 위험을 사전에 차단하였습니다.
* **과금 관리:** Make의 무료 계정 범위 및 OpenAI API 무료 테스트 크레딧 한도 내에서 완전하게 독립 작동하도록 엔지니어링하여 불필요한 자동 누적 과금 리스크를 원천적으로 방지하였습니다.

## 4. 보너스 과제 수행

### ① [보너스 1] 생성형 AI(OpenAI) 연동 자동화 성공
* 구글 시트의 키워드를 트리거로 받아 단순 복사가 아닌, OpenAI(ChatGPT) 액션을 결합하여 채널별 톤앤매너(블로그 '전략가', 인스타그램 '캐러셀 전략가')에 맞는 브랜드 스토리텔링 원고를 완전히 자동 생성하도록 파이프라인을 고도화함

### ② [보너스 2] 오류 대응 및 대체 경로(Gmail) 우회 전략 수립
* 구글 시트 업데이트 모듈의 데이터 매핑 및 행 번호(rowNumber) 바인딩 오류로 인한 데이터 유실 리스크를 감지한 후, 실행 실패 및 누수를 방지하기 위해 'Gmail 발송(Send an Email)' 액션으로 최종 결과물 전송 경로를 우회 설계함. 이를 통해 실행 결과물을 안전하게 메일함으로 증빙 및 적재하는 대체 경로 전략을 성공적으로 완수함
* **실패 알림 연계:** 위 우회 경로는 단순 대체 저장에 그치지 않고, 시트 업데이트 실패 시에도 결과물이 담당자(본인) Gmail로 즉시 도착하도록 설계되어 있어, 별도의 에러 알림 모듈 없이도 "실행 결과를 놓치지 않고 확인할 수 있다"는 점에서 실패 알림 기능을 사실상 대체함. 다만 이는 사후 확인(passive) 방식이며, Make의 Error Handler 라우트를 활용해 오류 발생 시점에 별도의 "실패 알림 이메일"을 즉시 발송하는 능동적(active) 알림 구조로 고도화할 수 있는 여지가 남아 있음.
